# API Integration — nockd Dashboard

How the dashboard talks to `nockd`. Grounded in **DESIGN.md** §5 (architecture), §6 (data
model), §9 (dashboard), §10 (security).

> **Status of this document.** The REST shapes below are a **proposed surface** consistent
> with DESIGN.md; the Control API in DESIGN.md is specified as "HTTP+JSON" without frozen
> routes. Align names/paths with the actual `nockd` Control API as it lands — the *fields*
> come straight from the §6 SQLite entities and should be stable. The dashboard and the CLI
> are both clients of this one API; the dashboard has **no privileged backdoor** (§9.1).

---

## 1. Transport & auth (§5.1, §9.3, §10)

- **Same binary, same origin.** `nockd serve` embeds the dashboard assets (`rust-embed`)
  and serves them from the same HTTP server that exposes the Control API. No separate web
  deploy, no CORS.
- **Local default:** the API's primary listener is a **Unix socket**, gated by file
  permissions. Browser access on the same box needs no extra config.
- **Remote is opt-in:** a TCP listener with **TLS + a bearer token** (generated by `nockd`
  on first run, shown once via CLI). When remote, send `Authorization: Bearer <token>` on
  every request and SSE connection.
- **Never** let the browser speak the NockApp **private/admin gRPC** (no TLS, no auth, §5.3
  / §10). `nockd` keeps each app's private gRPC bound to localhost, is the only speaker of
  it, and re-exports a safe, authenticated view through this API. The dashboard reads health
  and control **only** through `nockd`.

## 2. Data model (mirror of DESIGN.md §6)

The API serializes these entities. JSON field names are indicative.

```jsonc
// App (desired + observed merged for the UI)
{
  "name": "blackjack",
  "status": "running",            // running | degraded | stopped | crashing
  "desired_status": "running",    // running | stopped
  "artifact_hash": "blake3:a3f2…9c",
  "prev_artifact": "blake3:88c1…41",
  "endpoint_name": "mainnet-rpc",
  "restart_policy": "on-failure", // always | on-failure | never
  "restart_count": 0,
  "uptime_s": 1555200,
  "state_size_bytes": 148897792,
  "template": "http-server",
  "health": "serving",            // serving | degraded | down  (from private-gRPC probe)
  "chain_attach": "ok",           // ok | lagging | down
  "verified": "verified",         // verified | verifying | unverified | drift
  "created_at": "2026-06-04T16:08:00Z",
  "updated_at": "2026-06-22T14:02:00Z"
}

// Artifact (immutable, content-addressed)
{ "hash": "blake3:a3f2…9c", "created_at": "…", "hoonc_version": "0.9.2",
  "toolchain_pin": "rust 1.81 --locked", "source_manifest": "…",
  "provenance": { "build_host": "…", "reproducible": true, "verified": true } }

// Endpoint (named Nockchain attach target)
{ "name": "mainnet-rpc", "url": "http://1.2.3.4:5555", "kind": "remote", // remote | local-socket
  "reachable": true, "lag_ms": 240, "attached_apps": ["blackjack","wallet","treasury","name-registry"] }

// Secret — METADATA ONLY. The value never appears in any response.
{ "name": "blackjack-wallet", "used_by": "blackjack", "env_key": "WALLET_KEY",
  "created_at": "2026-06-04T…", "last_used": "2026-06-22T14:20:00Z" }

// Event (append-only audit/timeline)
{ "id": 4471, "ts": "2026-06-22T14:02:00Z", "app_name": "blackjack",
  "kind": "deploy",  // deploy|start|stop|crash|health|rollback|upgrade
  "detail": "blake3:a3f2…9c · health ✓ in 3.1s" }
```

> **Observed vs. desired (§6).** Live PIDs, current health, and resource metrics are held
> in memory by the supervisor (ephemeral, rebuilt on daemon restart), not in SQLite. The UI
> should treat status/health/metrics as live (SSE) and desired-state/artifacts/endpoints/
> secrets as durable (fetch + refresh on mutation).

## 3. Endpoints (proposed REST surface)

### Fleet & apps
| Method | Path | Returns / does |
|--------|------|----------------|
| `GET` | `/api/v1/apps` | `App[]` — Fleet table & tiles |
| `GET` | `/api/v1/apps/:name` | `App` + `{ current_artifact, prev_artifact, snapshots[], resources, attachment, secret_refs[] }` |
| `GET` | `/api/v1/apps/:name/events` | `Event[]` — App-detail timeline |
| `GET` | `/api/v1/apps/:name/metrics` | `{ cpu: number[], rss: number[], ts: string[] }` |
| `POST` | `/api/v1/apps/:name/restart` | restart per policy |
| `POST` | `/api/v1/apps/:name/stop` | set `desired_status=stopped` |
| `POST` | `/api/v1/apps/:name/deploy` | body: `{artifact_hash}` or multipart artifact upload → `{deploy_id}` |
| `POST` | `/api/v1/apps/:name/rollback` | re-point to `prev_artifact` (health-gated) → `{deploy_id}` |
| `POST` | `/api/v1/apps/:name/verify` | trigger reproducible rebuild-verify → `{verify_id}` |

### Endpoints / secrets / verification
| Method | Path | Returns / does |
|--------|------|----------------|
| `GET` | `/api/v1/endpoints` | `Endpoint[]` |
| `POST` | `/api/v1/endpoints` | `{name,url,kind}` |
| `PATCH`/`DELETE` | `/api/v1/endpoints/:name` | re-point / remove |
| `GET` | `/api/v1/secrets` | `SecretMeta[]` — **no values** |
| `POST` | `/api/v1/secrets` | `{name, value}` set/rotate (value write-only) |
| `GET` | `/api/v1/verification` | per-app verify status |

## 4. Live data over SSE (§9.2)

Server-pushed; this is what makes the static mocks "come alive." Use
`text/event-stream`. Suggested streams:

| Stream | Drives |
|--------|--------|
| `GET /api/v1/apps/:name/logs?follow=1` | App-detail **LIVE LOG** — append each line, autoscroll, show `● FOLLOWING` |
| `GET /api/v1/events?follow=1` | Fleet status pills + reconciler line; per-app timeline |
| `GET /api/v1/deploys/:id/events` | Deploy modal pipeline (see below) |
| `GET /api/v1/verifications/:id/events` | Verify progress bar |
| `GET /api/v1/endpoints?follow=1` | Endpoint reachability + lag bars |

**Log line shape:**
```jsonc
{ "ts": "14:22:08.114", "verb": "poke", "msg": "%ping ok", "dur_ms": 12 }
// verb ∈ poke|peek|chain|snap|…  → color per DESIGN-SYSTEM status/semantic palette
```

**Deploy event shape** (map onto the six pipeline rows):
```jsonc
{ "step": "health_gate",   // build|hash|upload|start|health_gate|retire_old
  "state": "active",       // done|active|pending|failed
  "detail": "Ping ok · gRPC health SERVING · poke %ping…",
  "progress": 0.72 }
```
The **health gate** is load-bearing (§8.1): the old instance is retired only after the new
one passes. On `state:"failed"` at `health_gate`, render step 5 red (▼) "GATE FAILED — old
instance retained" and stop — never mutate state on a failed deploy.

## 5. Security rules the UI must honor (§10)

- **Secrets:** never request, cache, log, or render a secret value. The list endpoint
  returns metadata only; the value field should not exist client-side. Always show the
  redaction bar. Rotation posts a value but never reads one back.
- **Redaction everywhere:** secret refs in App Detail show `→ ENV_KEY ••••`, not the value.
- **Token handling (remote):** keep the bearer token in memory, send over TLS only; never
  put it in a URL/query (SSE included — use a header or a short-lived ticket). Do not
  persist it to `localStorage`.
- **Audit:** every privileged action (deploy, rollback, stop, secret set, endpoint change)
  is appended to the event log by `nockd`; surface them in timelines so the dashboard is a
  faithful window onto that log.
- **Verification semantics:** `verified` = rebuilt-from-pinned-source hash match;
  `drift` = mismatch (tamper/out-of-band) → red; `unverified` = no source pin → grey;
  `verifying` = rebuild in progress → yellow + progress.

## 6. Packaging & front-end shape (§5.1, §9.2)

- **Embed the assets** in the binary (`rust-embed`) and serve from the same HTTP server —
  preserve the single-artifact, zero-install ethos. No CDN, no separate deploy.
- **Self-host the two fonts** (Jost, IBM Plex Mono) inside the bundle so the dashboard works
  fully offline.
- **Keep it modest** — "a small SPA or server-rendered + sprinkles … a window onto the API,
  not a second application with its own state of record." The design needs no heavy
  framework; the only real runtime concerns are SSE subscriptions and a tiny client router
  for the six views + deploy modal.
- **No client state of record.** Desired state lives in `nockd`'s SQLite; the UI fetches and
  reacts. After any mutation, re-fetch the affected resource rather than mutating local
  caches optimistically (except trivial toggles like table↔tile).

## 7. Status → color quick map (for wiring)

| API value | Glyph / color |
|-----------|---------------|
| `status: running` / `chain_attach: ok` / `reachable: true` / `verified` | ● blue |
| `status: degraded` / `chain_attach: lagging` / lag over threshold / `verifying` | ▲ yellow |
| `status: crashing` / `chain_attach: down` / `reachable: false` / `drift` | ▼ red |
| `status: stopped` / `unverified` / idle | ■ ink / muted |

See [`DESIGN-SYSTEM.md`](./DESIGN-SYSTEM.md) §4 for glyph construction and exact hexes.
